# 行政單位別名與官方知識文件設計

## 背景

目前系統已將學術單位的常用縮寫納入公告關鍵字展開、聊天提示詞與官方知識文件。行政單位也有相同的使用情境：使用者可能以「計網中心」、「研發處」或「育成中心」等簡稱提問，但官方頁面使用完整單位名稱。若沒有對應規則，聊天模型可能無法找到正確的官方資料，或將簡稱解讀成錯誤單位。

## 目標

1. 將使用者提供的行政單位別名對應到國立屏東大學官方完整名稱。
2. 將官方行政單位清單建立為可匯入 RAG 的 Markdown 文件與 YAML metadata。
3. 讓聊天規則在遇到行政單位名稱或別名時，使用 `search_documents` 並以資料庫中的官方文件作為來源。
4. 以自動化測試保障每個別名、提示詞規則與官方文件 metadata。

## 不在範圍內

- 不新增資料庫 schema 或 Migration；本功能使用既有官方文件匯入流程。
- 不新增 Chrome 權限或收集任何個人校務資料。
- 不把別名改造成所有 RAG 查詢的全域前處理器。此次與既有學術單位規則一致：別名設定供公告關鍵字搜尋展開，聊天端透過 system prompt 引導模型查詢官方文件。
- 不自行產生來源 URL；官方文件 metadata 只使用使用者提供並核對過的國立屏東大學行政單位頁面 URL。

## 方案

### 1. 公告關鍵字別名設定

在 `data/sources/announcements.yaml` 既有的 `keyword_search.aliases` 中加入行政單位別名。既有 `KeywordAliasResolver` 會以最長別名優先進行單次替換，因此不需新增 resolver 或修改搜尋流程。重複的「研發處」只保留一筆。

行政單位對應如下：

- 職推處＝職涯發展暨教育推廣處
- 研發處＝研究發展處
- 生輔組＝生活輔導組
- 衛生組、衛保組＝衛生保健組
- 軍訓室、軍安中心＝軍訓暨校安中心
- 生動組＝學生活動發展組
- 學諮中心＝學生諮商中心
- 原資中心＝原住民族學生資源中心
- 法制組＝行政法制組
- 校發組＝校務發展組
- 校研中心＝校務研究中心
- 校友組＝校友服務組
- 技合組＝技術合作組
- 學發組＝學術發展組
- 育成中心＝創新育成中心
- 國際處＝國際事務處
- 國合組＝國際合作組
- 國文組、外生組＝國際學生組
- 大陸組＝大陸事務組
- 職輔組＝職涯輔導組

- 場館組＝場館營運組

### 2. 聊天規則

在 `services/api/src/nptu_assistant/rag/prompts.py` 補充行政單位別名規則，並要求使用者詢問行政單位、處室、組、中心或室時使用 `search_documents`。回答必須以匯入資料庫的官方行政單位文件為來源；若查無資料，明確表示資料不足，不可自行推測。

### 3. 官方行政單位知識文件

新增：

- `data/official-documents/nptu-administrative-units.md`
- `data/official-documents/nptu-administrative-units.yaml`

Markdown 文件依官方頁面的行政單位層級記錄以下內容：校長室、秘書室、教務處、學務處、總務處、研究發展處、國際事務處、職涯發展暨教育推廣處、主計室、人事室、圖書館、計算機與網路中心及體育室，以及各單位頁面列出的所屬組、室與中心。

YAML metadata 使用：

- `source_url`: `https://www.nptu.edu.tw/p/412-1000-86.php?Lang=zh-tw`
- `unit`: `國立屏東大學`
- `document_type`: `administrative_units_snapshot`
- `published_at` 與 `version`: `2026-07-13`

## 資料流

1. 使用者輸入行政單位簡稱。
2. 公告關鍵字搜尋使用 alias map 展開原始簡稱與官方完整名稱，並將完整名稱作為 retrieval query。
3. 聊天模型依 system prompt 將行政單位問題導向 `search_documents`。
4. RAG 從已匯入資料庫的官方行政單位文件取得內容與資料庫來源 URL。
5. 若沒有足夠的官方資料，回答資料不足，不生成未存在於資料庫的 URL。

## 測試與驗收

- `test_keyword_search.py` 以參數化測試覆蓋所有行政單位別名，確認每個別名可正規化為指定官方名稱。
- `test_rag.py` 確認行政單位問題會要求使用 `search_documents`，且提示詞包含代表性別名規則。
- 新增行政單位知識文件測試，確認 metadata 的官方來源、文件類型與主要行政單位及所屬單位都存在。
- 執行既有 API 測試與 `git diff --check`。
- 在 API 容器可用時匯入新文件，確認文件成功建立、分塊且資料庫來源 URL 正確。

## 風險與處理

- 「國文組」可能與學術單位語境混淆，但本次需求明確指定其對應「國際學生組」，因此只在行政單位別名規則中照指定保留。
- 官方頁面未列出使用者提供的簡稱，別名只作為本地對應規則，不會被寫入官方文件內容。
- 若文件匯入或搜尋無結果，沿用既有資料不足與查無資料的回答規則，不以模型推測補足。
